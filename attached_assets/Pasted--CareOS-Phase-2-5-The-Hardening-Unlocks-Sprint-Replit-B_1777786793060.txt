## 3. Pillar B — Silent Failure Hardening

Every external service can fail. Currently most fail silently (degraded gracefully). For demo and production credibility, every failure mode needs visible, recoverable UX.

### 3.1 Service status panel
New admin page at `/admin/system-health`:
- Live status of every external service: Anthropic, Textract, Twilio, Resend, Pusher, Maps, Web Push, Redis, S3/Object Storage
- Last successful invocation timestamp per service
- Error rate over last 24 hours
- "Test connection" button per service
- Visible to OWNER role only

### 3.2 Failed AI run inspection
Every AI agent invocation already logs to `agent_runs`. Build the UI:
- New page at `/admin/jobs/agent-runs`
- Filterable list of all AI runs with status (SUCCESS / FAILED / LOW_CONFIDENCE / TIMEOUT)
- Click a failed run → see input, prompt version, model response, error
- "Retry" button on failed runs
- Cost-per-agent breakdown over selectable time ranges

### 3.3 Dead-letter queue UI
BullMQ failed jobs should be visible to admin:
- Currently BullBoard at `/admin/jobs` shows queues, but failed jobs need clear retry/discard workflow
- Add bulk retry, bulk discard
- Alert email to OWNER when DLQ depth exceeds threshold (default: 10)

### 3.4 Webhook failure alerting
Twilio webhooks, Resend bounce webhooks, push notification deliverability failures — all need to surface to admin:
- New `webhook_events` table logging every inbound webhook with response status
- New `notification_deliveries` table tracking SMS/email/push outcomes per recipient
- Compliance alert generated if a critical notification (e.g., shift change to caregiver) fails delivery on all configured channels

### 3.5 Graceful degradation messaging
When a service is down, the user should see WHY a feature isn't working, not silence:
- Referral parser uploaded with Textract down → "AI parsing temporarily unavailable. Document saved, will auto-process when service recovers."
- Push notification disabled → "Browser push not enabled. We'll send via email instead."
- Maps API quota exhausted → "Drive time estimates temporarily unavailable. Distance shown only."

## 5. Pillar D — Test Coverage for Critical Paths

These tests must exist before any production agency runs on the platform.

### 5.1 Labor rule engine — hard-coded edge cases
Automated tests covering:
- CA Domestic Worker: 9.5hr → 9 reg + 0.5 OT
- CA Domestic Worker: 13hr → 9 reg + 3 OT + 1 DT
- CA Domestic Worker: 7-day consecutive → 7th day all OT until 8hrs, then DT
- Federal FLSA: 45hr week, 8hr days → 40 reg + 5 OT
- Federal FLSA: 50hr week, one 14hr day → 40 reg + 10 OT (no daily OT under FLSA)
- NY Wage Parity: 40hr week with home care exemption application
- TX FLSA-only: identical to federal
- Authorization drawdown race condition: two concurrent shift assignments to same auth — only one succeeds, the other gets a conflict error

### 5.2 Multi-tenant isolation tests
- Simulate 3 agencies, each with overlapping client/caregiver IDs
- Every endpoint tested for cross-tenant data leakage
- Prisma middleware verified to throw on unscoped queries

### 5.3 AI pipeline tests
- Referral parser: 5 mock VA PDFs (varying length, quality, format), verify field extraction accuracy >85% on demographics, auth number, expiration date
- Document classifier: 10-document fixture, verify 9+ correctly classified
- Schedule optimizer: deterministic test fixture, verify ranking output matches expected order
- Anomaly detector: seed 5 anomaly patterns, verify all detected within one cron run
- Care plan drafter: given an authorization fixture, verify drafted plan contains expected core tasks

### 5.4 Caregiver PWA offline sync test
- Headless browser, disable network, complete full visit lifecycle
- Re-enable network, verify all events sync with original timestamps
- Verify temp visit IDs correctly rewritten

### 5.5 IVR security tests
- Spoofed caller ID with valid PIN: works (caller ID isn't auth)
- Spoofed caller ID with wrong PIN, 3 attempts: hang up
- 5 wrong PINs across 15 minutes: caregiver locked
- 8 wrong PINs from one number: number locked
- Twilio signature validation: malformed signature rejected with 403
